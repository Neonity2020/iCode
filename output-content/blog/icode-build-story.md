# 我是如何构建 iCode 的

如果只用一句话描述 iCode，我会说它是一个面向 Codex 的桌面工作台。

但这句话背后，其实是我对“AI 编程工具应该怎么嵌入真实开发流程”的一连串思考。iCode 不是从“我要做一个更炫的界面”开始的，而是从一个很实际的问题开始的：

当我真的在本地做开发时，Codex 不应该只停留在一个命令行工具里。它应该和工作区、文件树、终端、审批、变更审阅、会话状态放在一起，成为我日常工作流的一部分。

## 起点：我想把 Codex 放进真实工作流里

最开始我面对的是一种很典型的割裂感。

在终端里跑 Codex，可以很快发起任务，但你会很快碰到这些问题：

- 任务在终端里发起，但文件操作在编辑器里做
- 变更在终端里生成，但审阅又要切回别的工具
- 需要授权、确认、查看状态时，信息分散在不同地方
- 如果有多个任务或多个工作区，切换成本会越来越高

换句话说，Codex 很强，但它还不是一个“工作台”。

我真正想做的，是把它变成一个本地优先、围绕工作区组织的桌面应用。这个应用不是替代编辑器，而是把 Codex 的交互、任务上下文、文件变更和本地命令执行，收束到一个统一界面里。

这就是 iCode 的起点。

## 第一版：先把它做成一个轻量 Electron 应用

我最先做的决定其实很朴素：先让它跑起来。

所以 iCode 的第一层实现是一个轻量 Electron 桌面应用。原因很直接：

- 它天然适合接入本地文件系统
- 它可以直接管理 PTY 终端
- 它能和系统里的 Finder、外部链接、本地工作区打通
- 它适合做一个真正“贴着开发环境”的产品

桌面端负责本地能力，Codex CLI 则作为核心执行引擎。也就是说，iCode 不是自己重新造一个智能体系统，而是把已有的 Codex CLI 接到一个更适合人类工作的界面里。

现在仓库里的桌面端已经明确承担了这些能力：

- 本地工作区选择
- 文件树浏览
- PTY 终端
- Codex CLI app-server
- 审批与外部链接集成

这些功能都不是装饰，它们是为了让“发起任务、查看上下文、执行命令、审阅结果”这条链路真正闭环。

## 第二步：我意识到不能把产品逻辑绑死在 Electron 上

如果只是做一个 Electron 壳子，项目很快就会变得难维护。

真正棘手的问题不是 UI，而是“哪些逻辑应该依赖桌面环境，哪些逻辑应该是共享的”。

于是我把仓库重构成了一个 monorepo：

- `apps/desktop` 负责 Electron 和本地能力
- `marketing` 负责官网 Landing Page
- `packages/app` 负责共享的 React 产品层
- `packages/platform` 负责平台契约

这一步很关键。因为它让我能把产品逻辑从宿主环境里抽出来。

换句话说：

- 桌面端知道怎么连本地文件系统、PTY 和 Codex CLI
- Marketing 端负责官网落地页和项目介绍
- 共享包只关心产品本身应该如何工作

这样做的好处是，桌面产品和官网展示都能放在同一个仓库里维护，同时不把桌面工作台误装成一个尚不存在的浏览器版入口。

这也是我后来一直坚持的方向：平台能力分层，产品逻辑共享，宿主负责注入能力。

## 设计上的核心：平台契约

在架构上，我给 iCode 定了一个很简单但很重要的原则：

应用层不能直接依赖 Electron，也不能直接假定 `window.icode` 一定存在。

因此我把平台能力抽象成了一个统一契约。产品层通过 `PlatformProvider` 读取能力，而不是直接依赖宿主实现。

这意味着：

- 桌面端通过 preload 暴露能力
- Marketing 端当前是官网落地页，不承载工作台宿主能力
- 共享 UI 只依赖能力契约，不关心具体宿主

这件事看起来像工程组织问题，但实际影响的是产品的寿命。

因为一旦 UI 直接绑死在桌面宿主上，后面你再想做 Web、远程服务、轻量版、企业定制版，都会非常难拆。

我希望 iCode 从第一天起就是可扩展的，而不是只能在某一个环境里活着。

## 真正的产品逻辑：不是“聊天”，而是“工作区驱动的协作”

很多 AI 工具的默认范式是聊天窗口。

但我越来越确信，对于编程场景来说，聊天只是入口，真正的中心应该是工作区。

所以 iCode 的界面设计是围绕工作区组织的：

- 左侧是文件树和工作区导航
- 中间是对话、活动和任务流
- 右侧是文件变更、终端和附加信息
- 底部或侧边可以承载更细的控制项

这个结构不是为了复杂，而是为了让用户始终知道：

- 当前在哪个项目里
- 当前任务影响了哪些文件
- 当前需要哪些确认
- 当前终端在做什么
- 当前会话进展到哪一步

我不想把 Codex 做成一个“问答机器人”，我想把它做成一个真正能参与开发工作的助手。

## 过程中我做过的几件具体事

iCode 的演进不是一蹴而就的，它是一系列很具体的产品打磨。

### 1. 多标签工作区栏

当我开始真的在本地用它处理任务时，马上遇到一个现实问题：一个工作区不够用。

所以我加入了多标签工作区栏。这样我可以在多个目录之间切换，而不必反复重建上下文。

这类功能很小，但它非常接近真实开发场景。因为一个工程师的日常不是单线程的，任务、仓库、临时修复、实验分支总会交替出现。

### 2. 文件树右键菜单和持久化状态

文件树不是只为了展示目录，它是操作入口。

我后来补了右键菜单，也把标签页和展开状态做了持久化。这个决定很务实：用户不该每次启动都重新展开同样的目录树，也不该每次重新定位文件。

这类“微小但反复发生”的摩擦，往往比大功能更影响体验。

### 3. 工作区 git 变更显示

接着我把工作区 git 变更显示到侧边栏里。

这个能力非常重要，因为它把“AI 生成了什么改动”从抽象结果变成了可审阅的文件列表。用户不需要等到最后一刻才知道发生了什么，而是在任务过程中就能看到变更在怎么演化。

后来我又补了一个回退逻辑：如果不能直接拿到工作区变化，就通过 PTY 去读 `git status`。这说明一个产品在本地开发场景里必须有容错能力，不能把自己假设得太理想。

### 4. 粘贴图片支持

这个功能看上去不大，但很实用。

在真实工作里，你并不总是只有文本。有时候你需要贴截图、错误提示、界面参考图，甚至是产品原型。把图片作为消息附件支持进去，意味着 iCode 不再只适合纯代码任务，也能承载更丰富的上下文。

### 5. 设置面板和桌面图标恢复

随着功能逐渐变多，用户开始需要配置项：

- 默认模型
- Codex CLI 路径
- 终端 shell

我把设置 UI 重新补回来，也恢复了桌面图标。这些东西不像核心算法那样显眼，但它们决定了产品是不是“像一个真正可用的桌面应用”。

### 6. 终端体验打磨

最后我持续做了终端层的 polish。

原因很简单：对于开发者产品来说，终端不是附属品，而是主操作面之一。它要足够稳定、足够直观、足够贴合本地 shell 习惯，才能和任务流真正融为一体。

## 我为什么把官网 Landing Page 也做了出来

很多人会问：既然这是一个桌面应用，为什么还要做官网 Landing Page？

答案是，我不想让 iCode 的产品形态只停留在“一个 Electron 程序”。

官网 Landing Page 有两个作用：

- 它是项目对外的介绍窗口
- 它可以承载项目定位、能力说明和快速开始

更重要的是，我希望对外传播和桌面产品在同一个仓库里同步演进。当前 `marketing` 是 Landing Page，不是浏览器版工作台入口；这个边界写清楚，反而能让项目定位更准确。

从产品角度看，这是“一个桌面产品，一个对外窗口”；
从工程角度看，这是“一个仓库，清晰区分产品端和展示端”。

## 这个项目的架构原则，可以总结成三句话

第一，**本地能力必须可接入**。  
如果一个 AI 编程工具不能读本地文件、不能跑终端、不能处理工作区，它就很难真正进入开发者流程。

第二，**产品逻辑必须共享**。  
桌面端和官网展示端可以不同，但真正属于工作台的会话、状态、任务流、组件结构应该集中维护。

第三，**平台边界必须清晰**。  
宿主负责能力注入，产品层负责体验表达。边界越清楚，后面越容易演进。

## 做 iCode 之后，我最大的收获

这次构建 iCode 的过程，让我重新确认了一件事：

一个好的 AI 开发工具，不应该只追求“能回答问题”，而应该追求“能嵌入工作流”。

它要能理解工作区，要能展示变更，要能连接终端，要能处理审批，要能保留上下文，还要能让用户在任何时候知道自己处在什么状态。

换句话说，它不是一个聊天框，而是一个协作环境。

iCode 就是我朝这个方向做出的一个版本。

我不想把它做成一个停留在演示层面的工具，而是想把它慢慢打磨成一个真正能放进日常开发流程里的桌面工作台。

## 结尾

如果说这个项目最开始只是一个想法，那它现在已经变成了一套明确的产品方向：

- 面向 Codex
- 桌面优先
- 本地工作区驱动
- 官网 Landing Page 负责对外展示
- 用平台契约连接不同宿主

接下来我还会继续把它往更完整的工作台方向推进，让它不只是“能用”，而是真正“顺手”。
